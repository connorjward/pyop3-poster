\vspace{-1em}
\section{Mesh data $\neq$ ND arrays}
\vspace{-.8em}

N-dimensional (ND) arrays are the backbone of many areas of computational science. In particular they underpin machine learning frameworks such as PyTorch and JAX.

Their popularity as an abstraction is due to two factors:
\vspace{-1em}
\begin{enumerate}
	\item
		Hierarchically structured data is a natural programming model for many domains.
	\item
		Tensor operations can be compiled to extremely high performance code.
\end{enumerate}

However, ND arrays are not a suitable fit for calculations that involve unstructured meshes:
\vspace{-1em}
\begin{enumerate}
	\item
		Mesh data can be associated with different topological entities (e.g. cells and/or vertices, \cref{fig:sv_element}).
		This results in `ragged' dimensions in the data structure that cannot be represented using typical ND arrays (\cref{fig:dof_layout}).
	\item
		Mesh data are distributed in parallel and have complicated processes for ensuring global correctness.
\end{enumerate}

\myblue{\pyop3 is this missing abstraction.}
It is a library that provides a \myblue{numpy-like interface for interacting with unstructured mesh data}.
\myblue{Just-in-time compilation} is used to make the operations performant.
Just like PyTorch and JAX for ML, and Firedrake for FEM, \pyop3 provides a flexible, high-level, intuitive interface for building complex and performant programs, only this time for mesh-based simulation.

\vspace{-1em}
\section{The \pyop3 language}
\vspace{-.8em}

\pyop3 is a domain-specific language (DSL) that is embedded in Python.
Users express operations using a numpy-like syntax (e.g. \cref{listing:index}) that are just-in-time compiled to high-performance code (\cref{fig:codegen}).

\vspace{-.5em}
\begin{figure}[H]
	\centering
	\begin{minted}[fontsize=\small]{python}
  cell = mesh.cells.iter()
  vec[closure(cell)]["v"][::2]
	\end{minted}
	\caption{
		A non-trivial example of \pyop3 numpy-like syntax in action: accessing the even numbered vertices around a cell.
		\mintinline{python}{closure(cell)} returns all entities that touch the cell, and \mintinline{python}{"v"} extracts only the vertices.
		Note both the similarity between this code and typical ND array usage with numpy, and also how indexing operations may be chained together.
	}
	\label{listing:index}
\end{figure}

\vspace{-.5em}
\begin{figure}[H]
	\centering
	\includegraphics[width=\columnwidth]{codegen.pdf}
	\caption{Simplified example of the code generated from a \pyop3 loop expression.}
	\label{fig:codegen}
\end{figure}

Operations in \pyop3 are composable, and so it is possible to express non-trivial looping operations such as that shown in \cref{listing:patch}.

\begin{figure}[H]
	\centering
	\begin{minted}[autogobble,fontsize=\small]{python}
	loop(vertex := mesh.vertices.iter(),
	     loop(cell := star(vertex, k=2).iter(),
		  kernel(in_vec[closure(cell)], out_vec[vertex])))
	\end{minted}
	\includegraphics[width=.6\columnwidth]{patch.pdf}
	\caption{
		\pyop3 loop expression describing an operation over a vertex patch.
		\mintinline{python}{star(vertex, k=2)} will return all cells that touch the current vertex.
		Note that the inner loop over cells is `ragged' (has non-constant extent).
	}
	\label{listing:patch}
\end{figure}

\section{Representing mesh data}
\vspace{-.8em}

The core capability that ND arrays provide is the ability to address a block of memory using a set of indices (multi-index).
They do this via an \myblue{index expression} of the form
\begin{equation*}
	(i_0, i_1, \dots, i_n) \to \mathit{offset}
\end{equation*}
An ND array is composed of multiple \myblue{axes} and is addressed with an \myblue{index per axis}.

As an example, consider an array with shape \mintinline{text}{(3, 2)}.
It has 2 axes with extents 3 and 2, and has offset expression $2i_0+i_1$.

\columnbreak

%----------------------------------------------------------------------------------------
%	THIRD COLUMN
%----------------------------------------------------------------------------------------

To represent mesh data without losing important topological information, \pyop3 necessarily has a much richer abstraction for describing unstructured data layouts than simple ND arrays: the \myblue{axis tree}.

Like ND arrays, an axis tree is composed of multiple axes and has the ability to convert a multi-index into an offset.
However, in \pyop3, axes are composed of 1 or more \myblue{components}.
This means that axes are able to \myblue{branch}.

By permitting branching, axis trees can represent the heterogeneous types of data needed for unstructured mesh calculations.
For example, they can represent multi-field and multi-entity data layouts like that shown in \cref{fig:dof_layout} (see \cref{fig:axis_tree}) where the offset expressions include
\begin{equation*}
\begin{split}
	&([P_3]^2, v_2, n_0, d_1) \to 5 \\
	&(P_2^\mathrm{disc}, c_0, n_3, d_0) \to 24
\end{split}
\end{equation*}

\begin{figure}[H]
	\centering
	\begin{minted}[autogobble,fontsize=\small]{python}
	axes = AxisTree.from_nest({
	  Axis("space", {"u": 1, "p": 1}): [
	    {Axis("mesh", {"v": nverts, "e": nedges, "c": ncells): [
	      Axis("dof", 1), Axis("dof", 2), Axis("dof", 1),
	    ]},
	    Axis("mesh", {"v": nverts, "e": nedges, "c": ncells): [...]
	  ],
	})
	vec = op3.Dat.zeros(axes, dtype=float64)
	\end{minted}
	\caption{
		Simplified \pyop3 code constructing a vector (termed a `dat') with a data layout (axis tree) similar to that shown in \cref{fig:axis_tree}.
	}
	\label{listing:axis_tree}
\end{figure}

Axis trees underpin \pyop3's expressive syntax because complex indexing operations (e.g. \cref{listing:index,fig:codegen,listing:patch}) are simply axis tree transformations.

An additional benefit of axis trees is that they \myblue{decouple data semantics from implementation} - one can permute axes of the axis tree to achieve a semantically equivalent data structure with a different data layout.
