\vspace{-1em}
\section{Mesh data $\neq$ ND arrays}
\vspace{-.8em}

N-dimensional (ND) arrays are the backbone of many areas of computational science. In particular they underpin machine learning frameworks such as PyTorch and JAX.

Their popularity as an abstraction is due to two factors:
\vspace{-1em}
\begin{enumerate}
	\item
		It is a very natural programming model for many domains.
	\item
		Tensor operations may be compiled to extremely high performance code.
\end{enumerate}

Unfortunately, ND arrays are not a suitable fit for calculations that involve unstructured meshes:
\vspace{-1em}
\begin{enumerate}
	\item
		Mesh data can be associated with different topological entities (e.g. cells and/or vertices, \cref{fig:sv_element}).
		This results in 'ragged' dimensions in the data structure (\cref{fig:dof_layout}) that cannot be represented using typical ND arrays.
	\item
		Mesh data are distributed in parallel and have complicated processes for ensuring global correctness.
\end{enumerate}

\myblue{\pyop3 is this missing abstraction.}
It is a library that provides a \myblue{numpy-like interface for unstructured mesh data} that is compiled to fast code.
Just like PyTorch and JAX, \pyop3 will allow users to easily develop complex and performant numerical methods by composing operations at a convenient level of abstraction.

\vspace{-1em}
\section{The \pyop3 language}
\vspace{-.8em}

\pyop3 is a domain-specific language (DSL) that is embedded in Python.
Users express operations using a numpy-like syntax (\cref{listing:index}) that are just-in-time compiled to high-performance code (\cref{fig:codegen}).

\begin{figure}[H]
	\centering
	\begin{minted}[autogobble,fontsize=\small]{python}
		cell = mesh.cells.iter()
		vec[closure(cell)][0][::2]
	\end{minted}
	\caption{
		A non-trivial example of \pyop3 numpy-like syntax in action: accessing the even numbered vertices around a cell.
		\mintinline{python}{closure(cell)} returns all entities that touch the cell and indexing with \mintinline{python}{0} extracts only the vertices.
		Note both the similarity between this code and typical ND array usage with numpy, and also how indexing operations may be chained together.
	}
	\label{listing:index}
\end{figure}

\vspace{-1em}
\begin{figure}[H]
	\centering
	\includegraphics[width=\columnwidth]{codegen.pdf}
	\caption{Simplified example of the code generated from a \pyop3 loop expression.}
	\label{fig:codegen}
\end{figure}

Operations in \pyop3 are composable, and so it is possible to express non-trivial looping operations such as that shown in \cref{listing:patch}.

\begin{figure}[H]
	\centering
	\begin{minted}[autogobble,fontsize=\small]{python}
	loop(vertex := mesh.vertices.iter(),
	     loop(cell := star(vertex, k=2).iter(),
		  kernel(in_vec[closure(cell)], out_vec[vertex])))
	\end{minted}
	\includegraphics[width=.6\columnwidth]{patch.pdf}
	\caption{
		\pyop3 loop expression describing an operation over a vertex patch.
		\mintinline{python}{star(vertex, k=2)} will return all cells that touch the current vertex.
	}
	\label{listing:patch}
\end{figure}

\section{4. GPU support}

\begin{itemize}
	\item
		Just like many machine learning frameworks, \pyop3 is agnostic to the underlying hardware and in principle can generate code to execute on arbitrary platforms.
	\item
		Preliminary support for NVIDIA GPUs has already been merged.

\section{4. Future work}
\vspace{-.8em}

Once \pyop3 has been fully integrated into Firedrake there is scope for a variety of performance improvements and avenues of research to pursue.
Possible low-level changes include:

\vspace{-.8em}
\begin{itemize}
	\item Data layout transformations
	\item Loop tiling
\end{itemize}

\vspace{-.8em}
Whilst at a high level the following should also become expressible:

\vspace{-.8em}
\begin{itemize}
	\item Physics-based preconditioners
	\item Slope limiters
	\item Adaptivity
	\item Particle methods
	\item \myblue{And more!}
\end{itemize}
