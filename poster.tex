\section{FEM data $\neq$ ND-arrays}
\vspace{-.8em}

\begin{itemize}
	\item
		N-dimensional (ND) arrays are the backbone of many areas of computational science. In particular they underpin the entire field of machine learning.
	\item
		Why are they good? Nice abstraction, good to program with..., can optimise
	\item
		Data structures for the finite element method (FEM) do not conveniently fit into this paradigm.
		unstructured mesh... different topological entities... parallel..
	\item
		This prohibits the sort of development that we see in ML...
	\item
		\myblue{pyop3 is this missing abstraction...}
\end{itemize}

\vspace{-1em}
\section{1. Motivation}
\vspace{-.8em}

\begin{itemize}
	\item
		The application of compact computational kernels across a mesh (\cref{fig:fem_assembly}) crops up all the time in continuum mechanics simulations.
	\item
		Writing this traversal code by hand quickly becomes difficult due to the need to support different PDEs, discretisations, hardware, etc.
	\item
		\myblue{A software abstraction that can automate the construction of the traversal code is needed.}
\end{itemize}

\begin{figure}[H]
	\centering
	\includegraphics[width=.35\textwidth]{fem_assembly.pdf}
	\parbox{0.49\textwidth}{
		\caption{An example of applying a compact kernel to a mesh: finite element assembly of a matrix.}
		\label{fig:fem_assembly}
	}
\end{figure}

\vspace{-1em}
\section{2. Introducing \pyop3}
\vspace{-.8em}

\begin{itemize}
	\item
		\pyop3 is a new \myblue{domain-specific language} for automating calculations involving mesh traversal.
	\item
		It uses \myblue{code generation} (\cref{fig:codegen}) to outperform equivalent library interfaces.
	\item
		It is based on \pyop2~\cite{rathgeberPyOP2HighLevelFramework2012}, and work is in progress to replace \pyop2 with \pyop3 in the Firedrake finite element framework~\cite{FiredrakeUserManual}.
\end{itemize}

\section{3. A new abstraction for mesh data}
\vspace{-.8em}

\begin{itemize}
	\item
		To achieve this, \pyop3 necessarily has a much richer abstraction for describing unstructured data layouts than simple ND-arrays: the \emph{axis tree} (\cref{fig:axis_tree}).
	\item
		...Having a richer abstraction for data layouts makes composition easier, and thus \pyop3 is able to describe and generate code for a wider range of problems than \pyop2.
\end{itemize}

\section{4. GPU support}

\begin{itemize}
	\item
		Just like many machine learning frameworks, \pyop3 is agnostic to the underlying hardware and in principle can generate code to execute on arbitrary platforms.
	\item
		Preliminary support for NVIDIA GPUs has already been merged.
	\item
		Next steps are... (cite MLIR)
\end{itemize}

\vspace{-1em}
\section{4. Future work}
\vspace{-.8em}

Once \pyop3 has been fully integrated into Firedrake there is scope for a variety of performance improvements and avenues of research to pursue.
Possible low-level changes include:

\vspace{-.8em}
\begin{itemize}
	\item Data layout transformations
	\item Loop tiling
\end{itemize}

\vspace{-.8em}
Whilst at a high level the following should also become expressible:

\vspace{-.8em}
\begin{itemize}
	\item Physics-based preconditioners
	\item Slope limiters
	\item Adaptivity
	\item Particle methods
	\item \myblue{And more!}
\end{itemize}
